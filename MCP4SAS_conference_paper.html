MCP4SAS: An MCP Bridge for Running SAS Programs Through SASPy and Local SAS

Cheng Zhongshan
MCP4SAS Project
GitHub: https://github.com/chengzhongshan/MCP4SAS

Abstract

SAS remains a core analytic platform in clinical research, biomedical informatics, public health, regulated reporting, and many other data-intensive domains. At the same time, modern AI-assisted development environments are increasingly organized around tool protocols that let an agent submit jobs, inspect files, and iterate on code. MCP4SAS is a lightweight Model Context Protocol (MCP) package that connects these two worlds. It exposes SAS execution through SASPy and through a direct local batch helper, allowing AI agents and users to submit SAS code, run SAS files, upload and download SAS ODA files, inspect remote file metadata, list SAS ODA directories, reuse persistent SASPy sessions, and monitor long-running jobs without blocking the client.

The package wraps SASPy with a Perl command-line interface and a local Python session server for SAS ODA, local Linux SAS, local Windows SAS through SASPy IOM, and other SASPy configurations. It also ships a direct local SAS runner that can launch Linux SAS or Windows sas.exe without SASPy for one-shot batch jobs. The design emphasizes practical reproducibility: repo-local Python and Perl dependencies, cross-platform installers, SASPy configuration templates, persistent session reuse where SASPy supports it, explicit timeout controls, SAS macro loading from the SAS ODA ~/Macros directory, and diagnostic status files. This paper describes the motivation, architecture, installation model, major implementation details, example workflows, and lessons learned while building MCP4SAS as a standalone bridge for SAS automation in AI-assisted computing environments.

Keywords: SAS, SAS OnDemand for Academics, SASPy, local SAS, Model Context Protocol, MCP, AI agents, automation, reproducibility, Perl, Python

Introduction

SAS programs often live in workflows that are larger than a single editor or a single statistical task. A user may need to upload input files, submit a SAS program to a remote SAS environment, wait for a long computation to finish, download result artifacts, inspect logs, modify code, and repeat. In traditional interactive work this cycle is handled manually. In an AI-assisted workflow the same cycle needs to become a tool that an agent can call safely and repeatedly. The agent should be able to launch work, return control to the user while a job runs, inspect the status later, and provide enough diagnostic context when something fails.

MCP4SAS was created for this purpose. The project packages a SAS command-line runner together with an MCP HTTP server. The SASPy-backed runner gives direct shell access to SAS ODA, local Linux SAS, local Windows SAS through SASPy IOM, and other configured SASPy targets. A second direct helper runs local Linux SAS or Windows sas.exe without SASPy when the user needs a simple batch job rather than an interactive SASPy session. The server exposes these capabilities as MCP tools so that an AI agent can call SAS without needing to understand all shell quoting, temporary file, persistent session, and status-monitoring details. The package is intentionally focused on SAS; it is separated from larger project-specific pipelines so that it can be reused in many SAS programming contexts.

Motivation

Four practical issues motivated MCP4SAS. First, SAS ODA is remote, while the code authoring environment is local. File upload, file download, HTML result handling, authentication, and session reuse need to be coordinated. Second, sites may also have licensed local SAS installations on Linux or Windows, and a useful bridge should not force those users through SAS ODA. Third, SAS jobs may run longer than an AI agent's request timeout. A direct blocking call can make the agent appear stalled even when the SAS job is healthy. Fourth, SAS macro workflows often depend on a remote macro directory, and the called macro may itself call submacros. A robust bridge must make macro loading predictable without forcing users to manually include every dependency.

MCP4SAS addresses these issues with a layered design. The Perl runner provides a stable command-line interface. The Python session server manages persistent SASPy connections when SASPy is the selected target. A direct shell helper runs local SAS executables for one-shot batch jobs when SASPy is not desired. The MCP server wraps both execution paths in a background job model. Together these layers give both humans and AI agents a consistent way to run SAS code from local workspaces.

Overview of MCP4SAS

The repository contains seven main components. The first is server.pl, an HTTP MCP server exposing SAS tools. The second is run_sas_codes_or_files_in_ODA.pl, a command-line interface for running SAS code or SAS files through SASPy. The historical filename keeps the ODA wording for compatibility. The third is MCPDeps/SAS_ODA_Runner.pm, a Perl module that coordinates local execution, SASPy calls, file transfer, and macro preparation. The fourth is MCPDeps/sas_oda_session_server.py, a persistent local Python server that owns reusable SASPy sessions. The fifth is MCPDeps/importallmacros_ue.sas, a SAS helper used to load remote macros from the SAS ODA ~/Macros directory. The sixth is RunLocalSASDirectly.sh, a local batch helper that runs Linux SAS or Windows sas.exe without SASPy. The seventh is install/sascfg_personal.template.py, a SASPy configuration template for ODA, local Linux SAS, local Windows SAS, and remote IOM targets.

Component Role
server.pl Provides the MCP endpoint and exposes SASPy-backed and direct local SAS tools to AI clients.
run_sas_codes_or_files_in_ODA.pl Runs SAS code, SAS files, uploads, downloads, deletes, listings, login checks, and SASPy connection checks from the command line.
MCPDeps/SAS_ODA_Runner.pm Implements the Perl control layer, timeout management, macro preparation, file operations, and result handling.
MCPDeps/sas_oda_session_server.py Maintains persistent SASPy sessions so repeated submissions do not repeatedly initialize the full SAS connection.
RunLocalSASDirectly.sh Runs local Linux SAS or Windows sas.exe directly as a one-shot batch job without SASPy.
install/sascfg_personal.template.py Provides example SASPy configurations for ODA, local Linux SAS, local Windows SAS, and IOM servers.
MCPDeps/importallmacros_ue.sas Loads macro code from SAS ODA ~/Macros for macro-heavy SAS programs.

Architecture

Figure 1 summarizes the architecture. The user or AI agent sends a request to the MCP server. The MCP server creates a background worker and then chooses one of two execution paths. For SASPy-backed work, it invokes the Perl runner, which starts or reuses a local Python session server. The Python server owns the SASPy session that connects to SAS ODA, local Linux SAS, local Windows SAS through IOM, or another configured SASPy target. For direct local work, the MCP server invokes RunLocalSASDirectly.sh, which starts a local SAS executable as a batch process without SASPy. In both paths, logs and result artifacts are written to local diagnostic files that can be read later by either a user or the agent.

Figure 1. MCP4SAS execution flow from MCP client to SAS execution targets.

AI client or user | v server.pl MCP endpoint | v background worker | +-- SASPy path: | run_sas_codes_or_files_in_ODA.pl | MCPDeps/SAS_ODA_Runner.pm | local Python SASPy session server | SAS ODA, local Linux SAS, local Windows SAS, or IOM | +-- direct local path: RunLocalSASDirectly.sh local Linux SAS or Windows sas.exe batch process

This design isolates long-running SAS work from the MCP request lifecycle. The first MCP call starts work and returns a process identifier (PID). A later MCP call with the PID checks whether the background worker is still running. If it is complete, the MCP server returns the saved diagnostic output. This pattern is important for AI systems that should not spin in tight polling loops or keep a single request open for the full duration of a SAS job.

Command-Line Runner

The SASPy-backed command-line runner supports SAS code, SAS files, standard input, persistent sessions, login validation, SASPy connection validation, file upload, download, delete, remote file inspection, and remote directory listing. A minimal SAS ODA command is shown below.

OPEN_RESULT=0 ./run_sas_codes_or_files_in_ODA.pl \ --code "proc print data=sashelp.class(obs=5); run;" \ --persistent \ --session-id demo

The same runner can execute a SAS file:

OPEN_RESULT=0 ./run_sas_codes_or_files_in_ODA.pl \ --file my_program.sas \ --persistent \ --session-id demo

For code containing raw data blocks, standard input avoids shell quoting problems:

printf '%s\n' \ 'data a;' \ 'input a @@;' \ 'datalines;' \ '10 20' \ ';' \ 'run;' \ 'proc print data=a;' \ 'run;' | OPEN_RESULT=0 ./run_sas_codes_or_files_in_ODA.pl \ --code - \ --persistent \ --session-id demo

SASPy Target Configuration

MCP4SAS now uses SASPy configuration names to select the SAS target. The default target is oda, but local SAS users can choose a different configuration explicitly. The repository includes install/sascfg_personal.template.py and install/install_saspy_config.sh so that first-time users can start from a working configuration template rather than building a SASPy file from scratch.

bash install/install_saspy_config.sh SASPY_CFGNAME=oda ./run_sas_codes_or_files_in_ODA.pl --check-sas-oda-login-only SASPY_CFGNAME=linuxlocal \ MCP4SAS_LOCAL_SAS_PATH=/opt/sasinside/SASHome/SASFoundation/9.4/bin/sas_u8 \ ./run_sas_codes_or_files_in_ODA.pl --check-saspy-connection-only SASPY_CFGNAME=winlocal ./run_sas_codes_or_files_in_ODA.pl --check-saspy-connection-only

The --check-saspy-connection-only option validates the selected SASPy configuration with a small PROC SETINIT submit. ODA-specific credential prompts are restricted to the oda configuration so that local SAS users are not asked for SAS ODA credentials by mistake.

Direct Local SAS Without SASPy

Some users have a licensed SAS installation on the same Linux or Windows host and want the MCP server to run it directly, without SASPy. MCP4SAS supports this case with RunLocalSASDirectly.sh and the MCP tool run_local_sas_without_saspy. The helper locates a SAS executable from environment variables or standard installation paths and submits the SAS program as a batch job.

export MCP4SAS_LOCAL_SAS_EXE=/opt/sasinside/SASHome/SASFoundation/9.4/bin/sas_u8 export MCP4SAS_WINDOWS_SAS_EXE='/cygdrive/c/Program Files/SASHome/SASFoundation/9.4/sas.exe'

This direct path is intentionally different from the SASPy path. It is useful for one-shot local batch jobs and for environments where SASPy is not available or not desired. It does not provide persistent session reuse, because each tool call starts a fresh SAS process. As a result, WORK data sets, macro variables, librefs, options, and loaded macros do not carry over to the next tool call. Users who need persistent local SAS state should use a SASPy configuration such as linuxlocal or winlocal.

MCP Server Interface

The MCP server is started with:

perl server.pl daemon -m production -l http://127.0.0.1:8080

The MCP endpoint is:

http://127.0.0.1:8080/mcp

The server exposes four tool names:

  • run_sas_codes_or_files_in_ODA
  • run_sas_codes_or_script_in_ODA, a compatibility alias
  • run_local_sas_without_saspy
  • run_sas_codes_or_script_on_local_Windows, a compatibility alias for the direct local SAS tool

A typical MCP tool argument payload is:

{ "sas_codes_or_file": "proc print data=sashelp.class(obs=5); run;", "persistent": "1", "session_id": "demo" }

For long jobs, the first MCP call returns a PID and an output file. The client can poll later with:

{ "pid": 12345 }

Configuring AI Agents

The most important operational step is to start server.pl before starting the AI agent. MCP4SAS should normally be bound only to the loopback interface:

perl server.pl daemon -m production -l http://127.0.0.1:8080

All MCP-capable clients then point to the same local endpoint:

http://127.0.0.1:8080/mcp

OpenAI Codex

Codex supports MCP servers in both the command-line interface and IDE extension. For a Streamable HTTP MCP server such as MCP4SAS, users can add the endpoint with the Codex CLI:

codex mcp add mcp4sas --url http://127.0.0.1:8080/mcp

The same configuration can be written manually in ~/.codex/config.toml or, for a trusted project, in .codex/config.toml:

[mcp_servers.mcp4sas] url = "http://127.0.0.1:8080/mcp" tool_timeout_sec = 120 default_tools_approval_mode = "prompt" enabled = true

After restarting Codex, the /mcp command in the terminal UI can be used to confirm that the mcp4sas server is available. A useful prompt is explicit about the session identifier and polling behavior:

Use the MCP4SAS tool to run: proc print data=sashelp.class(obs=5); run; Use persistent session_id demo. If the tool returns a PID, wait and poll later.

Gemini CLI

Gemini CLI commonly uses a project-level .gemini/settings.json file for MCP tool configuration. For MCP4SAS, the relevant entry is:

{ "mcpServers": { "mcp4sas": { "httpUrl": "http://127.0.0.1:8080/mcp" } } }

After changing this file, users restart gemini and ask it to use the MCP4SAS tool. Older Gemini CLI builds or other clients that only support stdio MCP servers may need an HTTP-to-stdio MCP proxy, or a newer build with HTTP MCP support.

Ollama and Local LLM Clients

Ollama is a local model runtime rather than an MCP client by itself. To use an Ollama-backed model with MCP4SAS, users run Ollama, run MCP4SAS, and then place both behind an agent shell, desktop client, or bridge that supports MCP tool calls:

ollama serve ollama pull qwen2.5-coder:7b perl server.pl daemon -m production -l http://127.0.0.1:8080

Different local clients use slightly different JSON keys. Some expect url, some expect httpUrl, and stdio-only clients need a proxy. A generic HTTP MCP configuration looks like:

{ "mcpServers": { "mcp4sas": { "url": "http://127.0.0.1:8080/mcp" } } }

Recommended Agent Workflow

The recommended workflow is: start server.pl; ask the AI agent to call run_sas_codes_or_files_in_ODA with SAS code, a SAS file, or a file-management request when the target is SASPy-backed; use a persistent session_id for related SASPy tasks; call run_local_sas_without_saspy for direct one-shot local Linux or Windows SAS batch jobs; if the response returns a PID, poll no more often than about every 30 seconds; and inspect the returned diagnostic file when SAS reports errors or unresolved macro warnings. This workflow lets long SAS jobs proceed without holding a single MCP request open for the entire runtime.

Persistent SAS Sessions

SAS ODA session startup can take several seconds, and macro bootstrap may take longer for users with many macros in ~/Macros. Local SASPy targets can also benefit from keeping a SAS session alive. MCP4SAS therefore supports persistent sessions with an explicit session identifier for SASPy-backed targets. The same session can be reused across multiple submissions, which also preserves SAS WORK datasets and loaded macro state.

Persistent sessions are implemented through a local Python server listening on the loopback interface. The Perl runner uses length-prefixed JSON messages to ask the server to create sessions, upload files, submit SAS code, download files, inspect remote files, and shut down. The session server records detailed log messages, and the Perl client prints wait heartbeats that include elapsed time and timeout state. A typical heartbeat looks like this:

Waiting for SAS ODA session server response while reading response header (elapsed=60s, timeout=3675s)...

This message means that the local Perl client is waiting for the local session server to answer. During macro bootstrap or a long SAS program, the server may already be active inside SAS ODA. The elapsed time makes the wait easier to interpret.

The direct local SAS tool does not participate in this persistent session model. It starts a new Linux SAS or Windows sas.exe batch process for each call. This limitation is intentional and is documented in the tool description so that AI agents do not assume that state from a previous direct batch job will remain available.

Macro Loading

SAS programmers often organize reusable macro code under the SAS ODA ~/Macros directory. A macro invoked by a program may call several other macros. Loading only the top-level macro can therefore fail with unresolved macro warnings. MCP4SAS addresses this by supporting a global importallmacros_ue bootstrap that loads the remote macro directory when non-built-in macro usage is detected.

The local importallmacros_ue.sas helper is checked against the remote copy in SAS ODA HOME. If the size and timestamp match, upload is skipped. If the local copy is different, MCP4SAS uploads it. This reduces unnecessary file transfer while still ensuring that helper edits are not hidden by stale remote files.

Useful macro-related environment variables include:

SAS_ODA_AUTOLOAD_MACROS=1 SAS_ODA_MACRO_BOOTSTRAP_TIMEOUT_SECONDS=420 SAS_ODA_CLIENT_HEARTBEAT_SECONDS=20

Long-Running Job Handling

Long SAS jobs are expected in real analytic workflows. MCP4SAS separates two timeout concepts. The first is the overall wrapper timeout, controlled by --run-timeout-seconds or SAS_ODA_RUN_TIMEOUT_SECONDS. The second is the lower-level persistent session server response timeout. By default, the persistent submit response wait follows the wrapper timeout so that a two-hour wrapper timeout also permits a two-hour SAS submit.

OPEN_RESULT=0 ./run_sas_codes_or_files_in_ODA.pl \ --file long_job.sas \ --run-timeout-seconds 7200 \ --persistent \ --session-id long_job

If users need an unbounded wrapper timeout, they can use:

OPEN_RESULT=0 ./run_sas_codes_or_files_in_ODA.pl \ --file very_long_job.sas \ --no-run-timeout \ --persistent \ --session-id long_job

When a session becomes wedged, users can stop local SAS ODA helper processes:

./run_sas_codes_or_files_in_ODA.pl --kill-saspy-sessions

Installation Across Platforms

MCP4SAS ships with platform-specific install scripts. Each script creates a repo-local Python virtual environment and installs the required Perl modules under the repository. This reduces conflicts with system-level Perl and Python installations. The install process can also create a repo-local sascfg_personal.py template that includes SAS ODA, local Linux SAS, local Windows SAS, and remote IOM examples.

Platform Command Notes
Ubuntu or Linux bash install/install_ubuntu.sh Installs Java, Perl, Python, build tools, SASPy, Pillow, and Mojolicious.
macOS bash install/install_macos.sh Uses Xcode Command Line Tools and Homebrew.
Windows with Cygwin bash install/install_cygwin.sh Uses Cygwin packages and a Unix-like shell.
Conda bash install/install_conda.sh Creates or reuses a Conda environment for Python, Perl, and Java support.
Vagrant vagrant up, then bash install/install_ubuntu.sh Provides a clean Ubuntu virtual machine for Windows or macOS hosts.

Example Use Cases

Interactive SAS learning

Students using SAS ODA can ask an AI assistant to generate a SAS program, submit it through MCP4SAS, inspect the log, and revise the program. The persistent session lets earlier WORK datasets remain available for later commands, which matches how many users learn and debug SAS interactively.

Licensed local SAS execution

Organizations with licensed local SAS installations can use MCP4SAS in two ways. If SASPy is configured for the local environment, users can select linuxlocal, winlocal, or another site-specific configuration and reuse persistent SASPy sessions. If SASPy is unavailable or a simple batch run is sufficient, users can call run_local_sas_without_saspy to run Linux SAS or Windows sas.exe directly.

Macro-heavy analytic projects

Research groups often keep a large library of SAS macros. MCP4SAS can load the remote macro directory, submit code that calls top-level macros, and preserve the macro environment in a named persistent session. This is useful when the AI agent needs to debug the submitted code but should not repeatedly reload the entire macro library.

Remote file management

MCP4SAS supports uploading local files, downloading results, deleting remote files, listing directories, and checking file metadata. These operations can be used alone or combined with SAS code execution. As a result, the bridge is not only a submit tool but also a lightweight SAS ODA file manager.

Reproducibility and Diagnostics

Each run writes output artifacts to a named output directory. Important diagnostics include:

  • output.html.info.txt, a debug file containing submitted code, dependency logs, SAS logs, and result information.
  • output.run.status.json, a machine-readable status sidecar that records current state, phase, messages, timestamps, and completion status.
  • Session server logs, useful for diagnosing persistent SASPy session behavior.

These files make the workflow more transparent. An AI agent can read the same diagnostic files a human would inspect, summarize the log, identify SAS errors or unresolved macro warnings, and suggest next steps.

Security Considerations

MCP4SAS is intended for local trusted use. The MCP server binds to 127.0.0.1 in the recommended command. Users should not expose the server directly to untrusted networks because it can run SAS code and perform file operations. SAS ODA credentials can be supplied interactively or through environment variables for first-run setup. After validation, credentials are stored in the SASPy authinfo location used by the local account.

As with any tool that executes code, users should review agent-generated SAS programs before running them against sensitive data. The package does not remove the need for normal governance around protected data, credential management, or regulated reporting.

Limitations

MCP4SAS depends on the availability and responsiveness of the selected SAS target. SAS ODA and remote IOM configurations depend on network access, Java IOM connectivity, and correct SASPy configuration. Remote macro bootstrap time depends on the size and complexity of the user's macro directory. Some SAS ODA connection failures may require restarting local SASPy helper processes or creating a fresh session identifier. Local Linux and Windows SAS support depends on a valid licensed SAS installation and correct executable paths.

The direct local SAS runner is a batch adapter, not an interactive SAS session manager. It can start a job and poll its PID, but it cannot preserve WORK data sets, macro variables, librefs, options, or loaded macros across tool calls. Persistent local SAS state should be implemented through SASPy when possible.

Future Work

Future development can extend MCP4SAS in several directions. A standard stdio MCP transport could complement the current HTTP endpoint. A richer status API could expose structured SAS log findings, such as errors, warnings, unresolved macros, and generated file paths. Additional authentication helpers could make first-run SASPy configuration easier on managed systems. A future resident local SAS session manager could provide persistent local SAS execution without SASPy, but this would require robust command framing, log boundary markers, interrupt handling, timeout handling, and process cleanup. Finally, container recipes could provide a fully reproducible Linux runtime for organizations that want to deploy the bridge in a standardized environment.

Conclusion

MCP4SAS provides a focused bridge between SAS and AI-assisted development environments. By combining a SASPy-backed command-line runner, persistent SASPy session server, direct local Linux/Windows SAS batch helper, macro-loading support, long-run status polling, cross-platform installers, and an MCP HTTP server, it makes SAS workflows accessible to both human shell users and AI agents. The result is a practical foundation for interactive SAS programming, remote SAS file management, local SAS batch execution, macro-heavy analysis, and reproducible agent-assisted SAS workflows.
